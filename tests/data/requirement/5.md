<a name="_toc168306022"></a>随着社会经济形势的发展和金融市场发生迅猛变化，客户在银行中发生了多类的业务往来，包括存款、理财产品、转账汇款、借记卡、信用卡等等，如何通过这些交易活动整合我行业务产品的优势，精确判断、识别和营销客户，不断满足客户需求，提升客户体验和客户粘度，是我行个人业务发展的关键。

根据市场的发展趋势，成熟的积分管理和奖励方案已成为银行提升营销水平的有效方法。积分营销以其核算标准、规则灵活、运营高效、使用便捷等多重特点，受到银行、商户和客户等多方的青睐。银行开展的积分管理业务，将以前没有计算积分且分开存放的各项银行交易活动，通过积分管理系统将产生的积分放在一起统一集中累计，并在使用环节中为客户提供便捷、高效的积分兑换方式，从而让客户切实体验到银行的优质产品和服务，提升客户体验和粘度，取得了显著的效果

为进一步提高我行业务管理水平，实现全行对私客户的忠诚度管理及维护，我行拟建设全行综合积分管理系统。

我行综合积分管理平台系统应体现“以客户为中心”的设计原则，以提高员工操作效率、具有完善风险防控机制、能高效准确进行管理分析功能作为设计标准，系统在底层功能架构上应以实现全行综合积分为设计原则，积分累计账户应以简洁、灵活为设计原则，必须基于客户而不是产品进行积分账户的设置和管理，简化业务人员账户维护工作。系统功能应满足我行未来几年的业务发展需求，具有良好的可拓展性及业务功能前瞻性。 

积分系统前期整体规划主要做积分产生和自有渠道的服务对接、电子银行渠道的对接，尽快将综合积分产品平台搭建运营起来；后期主要延伸丰富积分的应用和积分可使用的渠道的拓展以及和商户合作的线下POS渠道。

系统主要分为三部分：

1、积分产品需能够支持个人客户相关的积分活动；

2、能够根据不同业务场景及渠道交易进行赠分的活动。

3、积分消费部分：能够支持兑换行内的自有服务；能够实现积分系统与行内电子银行部分渠道的对接；能够与行内现有商城连通，积分可在商城查询且兑换礼品。
# <a name="_toc476067240"></a>**1.总体目标**
以客户为中心，以市场为导向，强调产品与服务创新，整合服务渠道，关注风险管理与控制，以适应“以客户为中心”的变革，使综合积分管理系统真正成为向客户提供广西北部湾银行品牌与价值的交易、服务平台。

借助先进的综合积分管理系统完成对我行对私客户的忠诚度管理，实现客户的忠诚度维护，降低客户的流失率。

通过全行综合积分管理系统的建立，实现针对不同客户的差异化忠诚度营销体系的建立及差异化礼品回馈方案的建立，以满足不同客户在营销体系上的差异化管理。
# <a name="_toc476067241"></a>**2 业务要求描述**
提供业务解决方案，满足但不限于以下功能。
## <a name="_toc476067242"></a>**2.1系统需求**
### <a name="_toc311300311"></a><a name="_toc476067243"></a>**2.1.1系统管理**
#### **2.1.1.1机构管理**
管理员可通过此功能模块实现积分系统中的机构维护和管理，包括机构增加、撤销、修改、查询。系统应支持多级机构管理。
#### **2.1.1.2业务部门管理**
积分管理覆盖的有权发行积分机构需能支持不限数量及层级数。管理员可以通过此功能灵活维护机构的业务部门，包括增、删、改、查。
#### **2.1.1.3角色权限管理**
系统支持根据实际业务需要和组织结构，设立不同的岗位角色，角色可以通过选择列表的方式来被赋予不同的系统功能使用权限。支持设定角色的启用和停用状态。系统提供对角色的增、删、改、查，及角色权限的赋予、修改、组合等功能。
#### **2.1.1.4用户管理**
通过用户管理功能实现对系统用户的新增、修改、删除、查询操作。 
#### **2.1.1.5权限管理**
通过权限管理来定义角色、用户的功能权限、数据权限、业务查询范围等。系统根据登录用户的角色所具有的功能权限对系统功能访问进行严格控制以及业务流程的审批配置化，保障系统使用的安全。
#### **2.1.1.6渠道管理**
系统管理员通过渠道管理对允许接入的渠道进行合法性定义，并通过此功能定义渠道所允许办理的联机交易类型。例如是否允许积分查询、积分消费等联机交易。
#### **2.1.1.7密码管理**
1、所有系统用户必须通过密码对系统进行访问，系统对用户密码进行管理和合法性校验，保证系统的使用安全。用户可对密码进行修改。 

2、系统需支持在首次登陆时强制修改初始密码， 并对密码的强度、长度进行校验，密码须以密文方式存储，保证密码具有一定的安全性。 
#### **2.1.1.8系统参数管理**
通过参数管理模块，系统管理员可以查看涉及到本行积分系统全局性的一些参数信息，同时也可以对参数编号、名称、参数值、参数生效日期等要素进行维护操作。 比如：积分有效期、系统积分计算时一次提交的交易笔数、积分的现金价值、并发积分任务数等。
#### **2.1.1.9工作流配置**
根据需要，系统的业务流程可以在工作流引擎配置中自定义审批流程信息，包括审批节点定义、流转规则的简单配置。
### <a name="_toc311300312"></a><a name="_toc476067244"></a>**2.1.2客户服务管理**
#### **2.1.2.1客户基本信息查询**
用户可根据客户号、客户名称、账号，卡号等查询客户的基本资料以及相关的综合积分信息。

对于客户基本资料，系统提供对外接口，可根据业务需要与相关系统进行数据交换，建立适合业务的客户信息。
#### **2.1.2.2客户积分账户信息查询**
系统用户可以查询到某一特定客户积分账户内的积分总额信息、积分账户余额信息、客户积分交易明细信息、积分产生的原始交易明细信息。
#### **2.1.2.3客户积分账单查询**
系统根据指定的日期，定期自动生成客户的积分账单，并可供打印和查询。
#### **2.1.2.4客户冻结管理**
用户可以针对行内客户的异常积分情况进行冻结操作，被冻结的客户无法在主动的进行积分的消费与产生。当异常情况排除后，系统可以还原该客户为正常状态。

### <a name="_toc476067245"></a>**2.1.3商户管理**
#### <a name="_toc396321538"></a>**2.1.3.1商户类型管理**
系统提供商户类型的增、删、改、查、关联商户的功能。通过在活动中指定不同的商户类型，实现积分消费权限的划分。
#### **2.1.3.2商户管理**
系统提供商户信息的增、删、改、查，商户信息新增后，作为商户管理的关联对象。同时支持设置商户和行内的积分兑换的比例、以及积分可兑换的百分比。

同时，该功能并非商户信息主要维护入口，主要通过联机交易接口与外部进行商户信息同步，在同步时，外部能够设定是否允许作为兑换商户类型的关联对象，一般默认为允许。
#### <a name="_toc396321535"></a>**2.1.3.3兑换流水查询**
该功能主要提供积分兑换交易的查询功能，包括积分兑换正常交易和积分兑换冲正交易，同时提供对查询记录的导出功能
### <a name="_toc271889648"></a><a name="_toc311300313"></a><a name="_toc476067246"></a>**2.1.4积分管理**
#### **2.1.4.1积分账户管理**
积分账户管理包括积分账户的开立、信息修改、停用、冻结、解冻、查询。账户号由系统根据规则自动生成。积分账户的设置数量需不受限制，根据综和积分的特性，积分账户需要支持时效型账户、永久型账户、滚动失效型账户、红利型账户。同时还应支持特殊需要的专属账户类型，专属账户的积分只能在指定范围权限下使用。

一个积分账户须可以根据业务需要，灵活定义支持关联多个积分活动、多个产品、多个渠道，也可是一对一的关联。积分账户应是综合型账户，而不是产品型账户。

系统应该提供客户积分账户的自动开立功能，并对积分账户进行相关的账户检查（包括总分、明细、一致性检查等），保证积分数据的准确性、一致性、合法性。

支持积分手工修改。客户积分账户，可以手工进行维护（增加、减少、冲销）积分明细，经过系统设置的审批流程后反映到积分账户上。
#### **2.1.4.2积分活动管理**
积分活动管理包括积分活动的新增、修改、查询。业务部门根据营销的目标，通过此功能定义积分活动，包括积分活动的描述、活动的时间、参与的机构部门、成本分摊的方式等。需支持灵活弹性不限制积分活动数量的定制及管理功能。积分活动可以由一个或多个具体的积分计划构成，单个积分活动下不限制积分计划的数量。

根据积分活动的特性，系统必须支持营销类活动、渠道交易和资产余额类积分活动。

积分活动定义须支持灵活定义活动的成本分摊模式。支持按固定比例分摊、按活动产生积分值进行分摊，包括总分支行之间的分摊、各部门间的分摊，参与机构间的分摊等方式，并支持任意定义分摊的比例。
##### **2.1.4.2.1营销类活动**
营销类活动包括根据客户的等级、资产规模、个性特征等客户信息以及配合银行的品牌宣传等进行的积分奖励活动；无需客户发生交易行为，根据客户的固定属性就可以获得赠分。
##### **2.1.4.2.2渠道交易类活动**
渠道交易类活动须支持包括签约、开户、消费、银行产品和服务购买使用、中间业务办理等涵盖所有业务条线的开展。
##### **2.1.4.2.3资产余额类活动**
资产余额类必须支持根据客户在银行的各类资产或产品的持有情况进行定义积分奖励活动。
#### **2.1.4.3积分活动封顶**
积分活动支持单活动封顶，单活动单客户封顶，单活动单交易封顶。
#### **2.1.4.4积分计划管理**
积分计划是对积分活动的进一步细化。一个积分活动下可包括多个具体的计划。该功能是用来为定义的积分活动创建多个可执行的计划，针对每个计划可以定义不同的执行规则和赠分规则，以达到市场营销的目的同时也可以为客户赠送积分，吸引更多的客户参与活动。

功能包括积分计划的新增、修改、浏览、详细查询、删除等。
#### **2.1.4.5积分执行条件规则管理**
针对每个积分计划，灵活定义符合执行条件的具体规则。 通过执行条件规则定义满足积分计划的各项要素、逻辑条件，包括参与的机构、客户范围、产品、合作商、时间、业务种类、介质种类、交易要素（金额、频度等）、渠道等等。只有符合这些规则的客户才属于积分计划的参与者。系统必须提供灵活的规则引擎，支持自由定义配置。系统需支持灵活弹性的积分执行条件增、删、改、查询操作。积分执行条件需包括条件分类、条件名称、逻辑条件和值等信息。以参数化方式维护。

`        `在活动的规则设置上，可以涵盖零售银行所有业务条线（如借记卡消费、支票、储蓄、基金、国债、第三方存管、理财、中间业务等）进行积分累计，因此积分条件设置模型要全而且灵活，在进行解决方案讲解时要能够提供贵公司所实施过的相关规则模型，这些模型要能够完整地覆盖客户信息、卡信息、交易信息、资料余额类信息以及累计信息。
#### **2.1.4.6定单式营销管理**
需能支持特殊积分条件规则管理，包括客户名单的文件导入等，能实现字段值定义时引入超长字符串。比如，联通公司要完成同行方的合作，对联通公司提供的一万个高端VIP客户（仅有1万个手机号码）赠送50000分，他们可以自由使用行方提供的积分网络进行礼品的兑换，联通公司可以将这一万个客户的手机号通过逗号间隔的文本文件的形式上传到系统中，进行规则的设置。
#### **2.1.4.7积分赠送规则管理**
该功能是定义满足活动执行规则条件后，具体如何给客户赠分的规则设置，并且实现累计（包括累计金额、累计次数和累计积分），将累计的积点或次数转换成积分的积分条件规则进行管理。并可以支持过渡户之间的相互转帐，累积的红利点数作为赠分规则来使用。

系统必须提供灵活的模型化的规则引擎，支持各种赠分方式的灵活定义，包括固定赠分，按消费金额赠分、按周期、按区间、按消费金额比例赠分、按余额赠分、按资产状况赠分、按交易笔数赠分、按交易时段频度赠分、按使用的产品或服务种类赠分等等。系统须支持根据不同的方式特性灵活定义赠送的积分数量。 

同时系统须支持根据客户参与活动的具体情况，对积极参与活动的客户进行额外的二次奖励赠分即红利赠分。奖励的条件可以灵活定义。
#### **2.1.4.8活动分摊比例设定**
根据行内不同的积分清算方式，可灵活选择
##### **2.1.4.8.1机构间分摊**
机构与机构之间可以通过设定某一活动的分摊比例，来共同承办某一积分活动，支出的相应费用，由各机构按照当初约定的比例来支付。
##### **2.1.4.8.2部门间分摊**
部门与部门之间可以通过设定某一活动的分摊比例，来共同承办某一积分活动，支出相应费用，由各部门按照当初约定的比例来支付。
#### **2.1.4.9积分活动试算**
为了帮助业务部门开展积分活动，对积分活动进行预测，以便测算营销的费用和活动的结果等，系统必须提供积分试算功能。通过使用历史数据，对准备开展的活动进行事先的试算，通过试算的结果了解活动将可能产生的积分总体情况，

系统支持手工上传相应的活动试算数据，主要功能第一是验证新建立的活动规则设置是否正确，活动规则设置是否严谨，第二 可以通过试算的结果分析活动的赠分规则是否符合业务营销的需求，达到营销的目标；第三是通过活动试算，可以预估出本次活动的成本供参考。

支持按照时间、机构、单个营销活动或者多个营销活动组等不同的维度，利用一段时间内历史数据进行测算。
#### **2.1.4.10积分活动审核**
新建立的积分活动，需要经过相关岗位人员的审核流程，才能进行正式的发布使用。活动审核管理员可以通过活动试算了解活动预期结果后，对积分活动进行审核，审核通过的积分活动才能够进行积分计算。
### <a name="_toc476067247"></a>**2.1.5积分应用管理**
积分的定价标准取决于积分在对应渠道所能兑换的商品或服务的价值。因此，对于全行的积分，其积分价值是固定不变且全行统一的，即标准积分。

系统需至少支持积分兑换等积分使用方式，积分兑换可支持积分实时扣减功能。
#### **2.1.5.1积分查询**
系统需支持在我行积分商城上，可查询客户的积分信息情况。
#### **2.1.5.2积分特殊赠分**
系统需支持按照特殊规则，根据外围系统加工好的结果数据，直接导入积分系统。
#### **2.1.5.3积分支付**
系统应支持客户积分可在第三方电商上直接使用，通过行内的网关系统，和第三方系统对接。且支持使用积分在我行积分商城上兑换礼品。
#### **2.1.5.4积分赠送**
系统应支持客户按照行内要求进行某项活动后，获得相应的积分赠送。
#### **2.1.5.5积分互换**
系统应支持客户积分可和航空公司里程、运营商电信积分互换。
#### **2.1.5.6积分展期**
系统支持在特殊情况下，可以对客户到期的积分进行展期处理。展期需要花费一定数量的积分，系统可以定义按固定数值或按展期数量的比例来收取。
#### **2.1.5.7现金购买积分**
系统应支持通过使用现金购买积分的功能，现金购买积分独立清算。
#### **2.1.5.8积分转赠**
系统提供不同客户之间积分转移的功能。客户通过渠道发起积分转移申请联机交易，后台人工对积分转移申请审核通过后，生成积分转移批量文件，日终由系统自动进行积分转移。积分转移需要花费一定数量的积分，系统可以定义按固定数值或按转移比例收取。
#### **2.1.5.9积分抵用**
系统应支持银行卡交易过程中自动使用积分抵扣交易手续费、增值服务费、抵扣部分交易金额等，同时支持实时扣减积分。根据行内的积分使用策略，动态支持积分或积分+现金等积分抵用的方式。
### <a name="_toc461027885"></a><a name="_toc476067248"></a>**2.1.6礼品管理**
系统支持礼品与订单的管理、维护与统计。能够针对某一礼品进行礼品信息、供货商信息的维护，并能发布该礼品的销售信息，管理该礼品的库存情况等。能够针对将要缺货或已缺货的礼品，及时的发出通知。能对客户购买的多件礼品中，针对某一件礼品，进行单独的退货，不能影响客户购买的其他礼品。
#### **2.1.6.1礼品维护**
系统支持对行内礼品进行维护及管理，礼品的类型可以定制。
#### **2.1.6.2礼品上下架管理**
系统支持对礼品进行上架。即在原有礼品属性的基础上提供积分定价，设定积分金额兑换比例等，同时还可以根据需要设定礼品库存的最低预警数量。礼品上架需要经过审批后才能正式发布和兑换。
#### **2.1.6.3礼品上下架审核**
系统应对礼品的上下架应进行审核，审核通过后，上下架才可生效。
#### **2.1.6.4礼品库存查询**
系统应支持按供货商统计每种礼品的数量。
#### **2.1.6.5礼品库存调整**
系统应支持对具体礼品的当前库存情况进行手工调整。
#### **2.1.6.6礼品库存预警**
系统应支持当某些礼品库存数量达到预先设置的最低预警数量时，能够向维护人员发出预警提示。
### <a name="_toc311300315"></a><a name="_toc461027886"></a><a name="_toc476067249"></a>**2.1.7客户订单管理**
系统需提供强大的订单处理能力，并提供丰富的订单操作接口以支持不同的操作渠道进行订单的管理操作。系统需确保银行与客户订单的账务处理的正确性与及时性。
#### **2.1.7.1订单生成**
系统需提供订单生成接口，可以用于手机银行、网银、电话银行、积分商城、积分POS、微信银行、第三方合作商等渠道来产生相应的订单并送入积分系统进行后续处理。
#### **2.1.7.2订单查询**
系统需要支持客户从各个渠道（手机银行、网银、电话银行、积分商城、积分POS、微信银行、第三方合作商等）进行订单信息的状态查询。
#### **2.1.7.3订单取消**
系统需支持客户订单提交后允许订单取消的机制，以及相应的账务处理。在进行取消订单交易涉及到现金部分时，要能够符合行内的安全标准接口，调用其现金冲销交易。
#### **2.1.7.4退货订单**
系统需支持供货商订单退货机制，并支持退货后积分系统对订单涉及的积分进行反冲，并将订单涉及的积分反馈到积分账户中。积分账户如不存在，系统需能自动创建。

若涉到到现金的退款时，要能够根据行内安全标准接口，调用其现金的回退交易，完成客户现金入账。

### <a name="_toc476067250"></a>**2.1.8账务清算**
#### **2.1.8.1商户账务清算管理**
系统需提供完整强大的清算功能，能够对不限数量的商户进行有效的账务清算处理及账务结算。系统每月根据各个商户或合作商户的积分交易，进行积分的账务清算处理，产生商户或合作商与银行间的清算报表。

该报表能够统计出，商户每月的兑换积分总额，现金总额；每月退货冲回的积分总额，现金总额；以及银行需要承担的积分成本。

统计规则：以商户为单位对“已下单”、“已退货”以及兑换类型为积分+现金的订单进行结算汇总。

如果是和第三方的电商作为一种特殊的商户，则需根据不同的清算周期，出具清算报表。
#### **2.1.8.2银行内部分支机构营销成本分摊**
由于客户的积分是综合积分，可能来源于多个业务部门或机构，系统每天根据当天客户的积分交易情况，进行积分的银行内部账务清算处理，产生出当天各机构的清算报表。
### <a name="_toc311300316"></a><a name="_toc476067251"></a>**2.1.9联机交易管理模块**
系统需具备联机交易及联机可执行的积分交易权限管理功能，由系统管理员维护联机交易及权限调度功能。
#### **2.1.9.1外围渠道安全管理**
外围渠道的系统包括手机银行、网银、电话银行、积分商城、积分POS、微信银行、第三方合作商等系统。系统支持管理员可以自由设置接入渠道等信息，保证渠道有效使用和管理及联机交易接口的安全性。外围渠道安全管理包含联机交易功能权限管理，联机交易调试管理。具体实现每个联机交易权限管理，包括是否有效以及联机交易能够被哪些渠道的系统所访问。
### <a name="_toc311300317"></a><a name="_toc476067252"></a>**2.1.10报表统计分析模块**
系统支持根据报表的设定条件进行查询，且可依预先设定好的样式对报表数据结果进行展示。报表模块并需提供报表打印和报表下载功能。系统需能提供的报表包括以下常用报表：
#### **2.1.10.1积分分数查询统计**
用户可以按照月汇总出某个机构和业务部门下面的所有活动的积分赠送（＋）和兑换（－）的情况。

报表目的：是为了按照机构、部门和积分活动来统计银行送出的积分和兑换的积分状况，以便衡量银行支出和收入。
#### **2.1.10.2交易对账明细报表**
积分系统和前置每日现金对账明细。在进行积分+现金的交易时，现金的部分需要同行里的核心业务系统进行交易。
#### **2.1.10.3对账情况统计表**
积分系统和前置每日现金对账汇总信息。如总交易笔数，交易金额。
#### **2.1.10.4积分调整统计表**
当客户某积分账户的积分进行过手工调整以后，系统会对操作进行保存，可以通过该报表来对这部分的手工操作进行查询统计汇总。
#### **2.1.10.5客户积分区间统计情况**
1\.用户可以根据设定的不同积分区间对客户积分分布情况进行统计，按照部门分类。区间统计保留，按统一证件号客户统计。

2\.报表目的：可以分段统计出，持有不同积分数量的概要人数情况，以便开展有针对性的活动。
#### **2.1.10.6商户清算报表**
综合积分系统针对每个商户所提供的订单信息，能够对不限数量的商户进行有效的账务清算处理及账务结算。同时在后续的积分活动中给予更大的积分鼓励，让商户真正感受到和银行合作可以带来的实际好处，加深和商户合作程度，商户更鼓励持卡人使用银行的卡片，为商户带来更大的客户回流的同时，拓展潜在客户，实现银行、商户和持卡人的三方共赢的局面。
#### **2.1.10.7商户对账差错清单**
按照自然月末统计在统计月内银行与商户处理积分兑换订单的差异。
#### **2.1.10.8季度机构消耗积分统计报表** 
根据本季度各分支机构下客户消耗积分的情况，出具各分支机构消耗积分统计报表。
#### **2.1.10.9异常积分统计**
由于消费冲账原因而导致的客户积分变为了负值，可以通过该报表进行查询和统计。
#### **2.1.10.10积分审计查询**
针对客户积分账户和流水表中的数据按照固定的一定的规则进行校验，将不符合正常规则的记录展示在报表中，例如前一笔处理后余额是否等于下一笔处理前余额。
#### **2.1.10.11积分审计查询**
针对客户积分账户和流水表中的数据按照固定的一定的规则进行校验，将不符合正常规则的记录展示在报表中，例如前一笔处理后余额是否等于下一笔处理前余额。
#### **2.1.10.12季度机构消耗积分统计报表** 
根据本季度各分支机构下客户消耗积分的情况，出具各分支机构消耗积分统计报表。
#### **2.1.10.13异常积分统计**
由于消费冲账原因而导致的客户积分变为了负值，可以通过该报表进行查询和统计。
#### **2.1.10.14供应商礼品订单统计**
用户可以按照月汇总出，客户对各供货商所属礼品订单和退单在各种状态下的数量。
#### **2.1.10.15供应商退货订单统计**
按照自然月末统计在统计月内客户退货或取消订单的明细。
#### **2.1.10.16积分+现金兑换交易统计**
统计积分+现金兑换交易的订单统计表。
#### **2.1.10.17供应商结算汇总表**
依供货商提供的礼品成本，在客户发生兑换之后，向供货商进行结算。
#### **2.1.10.18供货商对账差错清单**
按照自然月末统计在统计月内银行与供货商处理积分兑换订单的差异。
#### **2.1.10.19供应商绩效评估统计**
按照自然季度末统计在统计季度内供货商供货情况。包括订单量、缺货礼品数量、订单量新增退单量、退单量新增、成本结算额、消耗总积分等内容。

### <a name="_toc311300319"></a>  **<a name="_toc476067253"></a>2.1.11批处理作业调度与监控**
系统应提供批处理作业调度和监控模块，系统管理员通过该功能可以对批处理作业和任务进行全方位管理。调度和监控系统必需是独立的业务处理系统，在进行积分系统跑批时，不得影响积分业务及交易7\*24小时不间断运作的要求。具体功能要求包括但不限于：
#### **2.1.11.1作业监控**
系统提供对批处理作业及其下各子任务运行监控的功能，能够对系统中各批处理作业及子任务的运行过程进行监控，能够显示各作业及子任务的名称、运行开始时间、运行结束时间、运行时长、运行状态等信息。系统内各作业及子任务在设定启动时间应能自动调起运行，不需要人工进行干预。当批处理作业运行异常中断时，应提供手工启停批处理作业的功能，当故障解决后，支持手工启动被异常中断的任务并从断点处继续执行。
#### **2.1.11.2作业调度**
系统应提供直观的界面，对批处理中的所有作业进行调度和维护的功能，能够为作业设定作业名称、运行时间和频度、工作日、包含的子任务、各任务之间的前驱后继依赖关系等内容。

系统内的各批处理作业及子任务在设定的工作日（不是系统日期）根据预设时间和频度按照任务之间的前驱后继关系自动运行，无需人工干预。系统内的批处理作业包括但不限于：

1、积分数据批量调整；

2、交易明细数据导入处理；

3、客户、卡片信息数据导入处理；

4、商户数据导入处理；

5、积分活动执行处理(积分计算)；

6、积分系统报表生成；

7、批量数据导出处理；

8、礼品商户账户清算处理；

9、分支机构及部门的账户清算处理。

10、积分活动规则修改重跑。
#### **2.1.11.3日志监控**
系统应提供运行日志和错误日志监控功能。

运行日志应包括作业运行日志和任务运行日志。每运行一次作业或者任务，都会产生一条运行日志。任务和作业有隶属关系，任务的运行日志和作业的运行日志也有隶属关系。每条任务运行日志，都是隶属于某条作业运行日志。在运行日志里，要统计每次作业或者任务的运行时长以及错误信息条数。

错误日志是记录批处理的错误信息。在批处理运行的时候，要利用各种方式捕获异常，并记录到数据库，方便在批处理运行异常时，根据记录的错误日志来排查异常原因。
### <a name="_toc311300322"></a><a name="_toc476067254"></a>**2.1.12系统接口功能**
系统需支持直连及间连接其他系统的功能，包括连接前置系统，FTP文件服务器，短信平台等。并详细说明积分系统与网上银行、Call Center、自助设备、核心业务系统、贷记卡系统、商户、门户网站及短信平台等进行连接及积分交易及信息交互的方式。
### <a name="_toc311300323"></a><a name="_toc476067255"></a>**2.1.13扩展性需求**
积分营销活动的开展支持参数化设置，新活动的设置及旧有活动的参数变量设置灵活，并支持参数化的管理。
### <a name="_toc476067256"></a>**2.1.14 多法人模式**
`   `支持多法人系统模式,不同法人机构可共用同一套综合积分管理系统,并根据机构不同,区分不同核算机制及规则引擎。可避免重复建设系统。

# <a name="_toc290897660"></a><a name="_toc476067258"></a>**4系统非功能性需求**
## <a name="_toc290897661"></a><a name="_toc476067259"></a>**4.1基本需求**
1、必须满足以客户为中心，积分汇总到客户层；

2、必须支持全量、增量两种批处理方式加工各种数据源系统的下传数据。

3、必须满足全国逻辑集中的要求，系统必须支持多级机构、部门的设置、统计以及汇总等功能；应支持个性化定制的活动需求。
## <a name="_toc290897662"></a><a name="_toc476067260"></a>**4.2性能需求**
1、系统应支持7×24运行，并不影响积分系统相关的在线联机交易；

2、后端批量计算处理，必须满足积分计算、增量信息ETL、流程控管等的实际要求，应能支撑日常交易流水达到百万级，核心积分计算后端跑批任务应在2小时以内完成。
## <a name="_toc290897663"></a><a name="_toc476067261"></a>**4.3运行维护需求**
1、能够提供系统的开发环境、测试环境、准生产环境、生产环境的关系框架图，并进行必要的解释说明及各环境的推荐配置；

2、系统能够达到具有99.9% 的运行能力，即每日因故障或维护原因而造成的不可用时间控制在 15分钟以内；

3、后端批量处理过程，应能够提供直观的批量运行的过程管理界面，并允许从上次错误点重新开始作业的批处理任务；

4、能够提供积分系统在运行期间的，后端系统处理流程和管理中信的操作监控管理功能，能够据此来进行问题追踪；

5、系统提供必要的参数维护，能满足必要的生产环境维护。
## <a name="_toc290897664"></a><a name="_toc476067262"></a>**4.4可靠性需求**
系统必须提供较高的可靠性，主要要求：

1、批量数据导入时，单个文件格式错误不影响其他文件的正常导入；

2、批量数据导入时，单条数据的错误不影响其他正确数据的导入；

3、单个活动的调整，不影响其他活动的正常执行；

4、除了需求中特别要求外，单个部门的所有业务操作，不影响其他部门的业务执行；

5、与外部系统的数据信息交换，在系统范围内可靠。
## <a name="_toc290897665"></a><a name="_toc476067263"></a>**4.5兼容性需求**
系统软件不同版本应能向下兼容。
## <a name="_toc476067264"></a>**4.6故障处理需求**
1、对于批处理运行中出现的故障，系统能够在监控画面进行醒目提示并且可以通过短信提醒系统管理员。

2、对于常见故障，在系统用户手册中提供故障修复的参考说明。

3、系统提供完整、详细的操作日志（包括用户界面操作日志、联机交易日志、批处理日志等）。出现故障时，能通过日志进行追查。
## <a name="_toc476067265"></a>**4.7知识产权**
因履行本项目合同所产生的研究开发成果及其相关知识产权权利归属，按下列条款确定：

（1）研究开发成果产生的知识产权归买卖双方共有，包括但不限于申请、登记注册、使用、转让等；

（2）买方有权利用卖方按照本合同约定提供的研究开发成果，进行后续改进。由此产生的具有实质性或创造性技术进步特征的新的技术成果及其权利归属，由买方享有；

（3）本系统开发、编译、运行的使用(或依赖)的第三方产品应尽量采用开源产品，否则应在采购投标时应向招标方正式声明并列举所有闭源产品。投标方应向买方提供系统实施的所有源代码，包含但不限于本系统开发、编译、运行所必须（或依赖）的、属于投标方拥有的工具、开发包与底层平台源代码。
# <a name="_toc476067266"></a>**5技术要求描述**
提供技术解决方案，满足但不限于以下要求。
## <a name="_toc476067267"></a>**5.1总体要求**
系统设计遵循的性能要求：可靠性、安全性、可定制化、可扩展性、界面友好性、集成能力等；采取模块化设计思想，满足应用实施的灵活伸缩。体现但不限于以下方面：

1、采用主流的架构和开发语言；

2、提供具有成熟的技术平台进行快速开发支持；

3、使用参数化配置；

4、支持多渠道接入，对外提供丰富的接口和服务；

5、提供有效客户信息模型，充分描述客户各种属性之间的关系；
## <a name="_toc476067268"></a>**5.2系统运行环境要求**
系统要能够运行于开放的系统平台，没有对硬件的相关特殊要求，支持市场上各种常用的操作系统与数据库系统。
## <a name="_toc476067269"></a>**5.3安全性要求**
主要包括：

1、控制用户权限，具有审查、审计、追溯历史交易的能力；

2、做到不同的功能需要有不同层次的安全接入；

3、确保系统内部网络不受到外部恶意攻击；

4、确保数据在通讯的各个环节中的安全性与保密性；

5、确保数据安全、用户安全、网络安全等。
## <a name="_toc476067270"></a>**5.4 延展性要求**
预留技术接口及系统具有良好延展性，方便系统未来能够支持和主流第三方电商的打通，从原有的行内渠道拓展到其它的渠道；将礼品供货商订单信息，集中放到积分系统进行管理。能够支持和商户合作的线下POS渠道的开通。
